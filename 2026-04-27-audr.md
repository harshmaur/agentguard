## Problem Statement

Companies are rapidly adopting AI coding agents (Claude Code, Cursor, Codex CLI, Windsurf), MCP servers, custom skills, browser harnesses, and automation runtimes — but they have no good way to know what each agent can access, which configs are unsafe, and whether company policy is actually being followed. Permissions are scattered across machines, repos, dotfiles, CI configs, IDE settings, and team docs. CISOs cannot answer basic questions: which agents can read production secrets, which MCP servers are running across the company, which configs were installed from unknown sources, which repos allow autonomous edits without review.

Audr is a CLI scanner + policy engine + SaaS control plane that discovers AI-agent configurations across developer machines and repos, evaluates them against company rules, and clearly distinguishes what is enforced vs detected vs advisory. The product wedge is AI-agent-config posture management specifically — not generic OSS-vulnerability scanning (Snyk owns that), not broad cloud-posture (Wiz owns that).

## Status Quo

What CISOs and platform teams are doing today, badly:
- Manual security review of agent configs (slow, doesn't scale, not continuous).
- Shared Notion/Google Doc policies that no one enforces.
- Slack reminders that decay within a sprint.
- Outright bans on AI agents, which developers route around with personal accounts.
- One-off scripts written by platform/security engineers (built, abandoned, rebuilt).
- Dependence on agent-vendor warnings (incomplete, vendor-specific, not policy-aware).

Cost of the status quo: at least one engineer-week per quarter spent on ad-hoc audits, growing risk surface as MCP/skill ecosystems expand weekly, and a real chance of an incident (leaked prod secret to an MCP server, malicious skill installed, autonomous shell access misused) that becomes the security team's "we didn't have visibility" moment.

## Target User & Narrowest Wedge

**Buyer archetype:** Security-conscious CISO, VP Engineering, Head of Platform, or security lead at a 20-500 person software company where AI coding agents are already being used.

**User archetype:** Platform engineer, security engineer, DevEx engineer, or senior developer responsible for standardizing AI development tools.

**Narrowest paid wedge:** A single static binary — `audr scan` — that a CISO can run on a developer laptop or in CI and get an HTML/SARIF/JSON report of risky AI-agent configurations within 10 seconds of install. Free under BSL/FSL on GitHub. The paid SaaS layer aggregates findings across a fleet, distributes policy centrally, runs continuous scans, integrates with SIEM/Slack, and produces compliance artifacts.

**Why narrow wins:** Snyk and similar incumbents will eventually add AI-config scanning, but they are not native to the agent ecosystem (MCP, skills, agent permissions). Audr wins by being deeply native first, then extending into adjacent surfaces. Honesty about enforced-vs-advisory and BYOD-vs-company-owned posture turns the "this is hard" technical reality into trust positioning.

## Constraints

- **Solo founder, part-time.** Honest MVP timeline is **5-8 weeks part-time** (not 2-3). Shorter is achievable only if trust artifacts are trimmed or the founder switches to focused full-time work.
- **MVP platform scope: macOS + Linux only.** Windows (`%APPDATA%`, PowerShell profiles) deferred to v2 — most CISO-archetype targets are Mac/Linux dev fleets.
- **LinkedIn-first launch.** No paid ads, no SEO funnel — distribution is a founder post + CISO network outbound. The artifact must be screenshot-and-share friendly.
- **Trust paradox.** A security tool installed from a stranger on LinkedIn is itself a supply-chain risk. The binary must be self-attesting (public source, signed releases via cosign, SBOM, self-audit subcommand) or it does not get installed. **SLSA Level 2 is the v1 target; SLSA Level 3 deferred to v2** — fully hermetic builds are a 1-2 month side quest by themselves.
- **Offline-by-default.** Security buyers distrust phone-home tools. The MVP runs entirely locally; SaaS sync is opt-in.
- **CISOs do not approve random closed binaries fast.** OSS source is not a "nice to have" — it is the only path through the buying objection.

## Taxonomy: enforced / detectable / advisory

This three-tier rubric is referenced throughout the doc and rendered on every finding. Definitions:

- **enforced** — Audr can produce a non-zero exit code that fails CI, blocks a commit hook, or refuses to load a config. The customer can write a policy that *prevents* the violation from reaching production. Example: "MCP server `command: npx some-server` without a pinned version → CI fails."
- **detectable** — Audr reliably finds the violation but cannot prevent it. The customer must rely on workflow (review, alert, education) to act on the finding. Example: "Skill at `~/.claude/skills/foo` references shell tool with broad write access → alert in dashboard, no programmatic block."
- **advisory** — Audr *cannot reliably detect* the issue but the policy is documented as best practice. The product names what it cannot enforce so the customer is not surprised later. Example: "Prompt injection through tool output may bypass agent-side allowlists; runtime sandboxing is the real mitigation."

Every finding must carry one of these tags. The CISO sale depends on the customer trusting that the tag is honest.

**P3.** BYOD vs Company-Owned is a first-class product axis with two policy modes, two enforcement levels, eventually two SKUs.
- BYOD: developer sees full findings; company sees aggregate ("3 high-risk MCP configs in the fleet"), not per-machine details. Privacy-aware by design.
- Company-Owned: company gets full per-machine telemetry, enforcement via managed install / MDM / wrapper launcher.
- This is a real moat. Most competitors will quietly elide the distinction; Audr names it loudly.

**P4.** Honesty about enforced-vs-detected-vs-advisory is the trust positioning, not a weakness. Security buyers reward calibration; the enforcement-vs-detection matrix is a sales artifact.

**P5.** MVP runs on developer machines AND in CI/repo. The binary works for both modes. BYOD privacy split is named on the roadmap as v2 differentiator (after 3 design partners are onboarded with the simpler version).

**P6.** Pricing is subscription from day one, not one-shot audit. Tooling changes weekly; daily/scheduled scans are the unit of value. Optional paid first-scan deep-audit ($500-$1k) as a sales motion that converts into per-developer subscription. The free OSS CLI is the foothold that makes the SaaS sale reachable.

### Test strategy

**Framework:** stdlib `testing` + `github.com/stretchr/testify` (assertions + mocks). Golden tests use `goldie/v2`. Property tests use `gopter` for the redaction invariant.

**Test layout:**

```
audr/
├── internal/{pkg}/{pkg}_test.go       # unit, table-driven where possible
├── testdata/                          # real-world anonymized fixtures
│   ├── laptops/                       # synthetic dev-laptop layouts (good + dirty)
│   ├── mcp/                           # MCP config samples (clean, dirty, malformed)
│   ├── skills/                        # skill markdown samples
│   ├── workflows/                     # GitHub Actions samples
│   └── golden/                        # golden output snapshots (HTML, SARIF, JSON)
├── e2e/
│   ├── install_test.go                # fresh-container install simulation
│   ├── scan_test.go                   # full scan against testdata/laptops/dirty
│   ├── verify_test.go                 # signed + tampered binary fixtures
│   ├── selfaudit_test.go              # tamper check positive + negative
│   ├── sarif_github_test.go           # imports SARIF via `gh api` smoke
│   └── redaction_property_test.go     # 1000-fixture invariant: no secret leaks
└── benchmarks/
    └── scan_bench_test.go             # <10s on standard fixture
```

**Mandatory regression-rule tests (failing any of these blocks a release):**

- `e2e/redaction_property_test.go`: scan a fixture containing 1000 randomly-generated planted secrets, assert ZERO appear in HTML, SARIF, or JSON outputs. This is the highest-stakes test in the suite — if redaction breaks once on LinkedIn, the trust thesis is dead.
- `e2e/verify_test.go`: BOTH the positive (valid signature → exit 0) AND tampered (modified binary → exit non-zero with `TAMPERED` banner). A "verify always returns clean" silent bug is undetectable without the negative case.
- `e2e/selfaudit_test.go`: same shape as verify — positive AND tampered fixtures.
- `e2e/sarif_github_test.go`: emit SARIF, import via `gh api repos/.../code-scanning/sarifs`, assert no warnings. Validates the doc's week-2 spike continuously.
- `internal/scan/parse_error_test.go`: malformed YAML/JSON/MD inputs → scanner emits `parse-error` finding tagged advisory and continues. Never crashes.
- `benchmarks/scan_bench_test.go`: full scan of standard laptop fixture completes in <10s on GitHub Actions ubuntu-latest. Regression breaks the LinkedIn demo promise.

**Per-rule test discipline:** Every rule in `internal/rules/builtin/` ships with three table-driven cases: (a) positive — config that triggers it; (b) negative — config that doesn't; (c) edge — boundary case. No exceptions. Adding a rule means adding 3 test rows.

**Coverage target:** 90%+ on `internal/`. Every public function in `cmd/`, `internal/scan/`, `internal/finding/`, `internal/redact/`, `internal/verify/`, and `internal/selfaudit/` MUST have a test before merge.

### CLI / observability conventions

- **CLI framework:** [`spf13/cobra`](https://github.com/spf13/cobra) (idiomatic for Go CLIs; same as `gh`, `kubectl`, `helm`). Subcommands: `scan`, `verify`, `self-audit`, `update`, `version`.
- **Logging:** stdlib `log/slog` with a text handler by default and JSON handler for `--log-format json`. Default level WARN; `--verbose` lifts to INFO; `--debug` lifts to DEBUG. All structured (`slog.Info("scan-started", "files", n, "workers", w)`). Logs go to stderr; report output goes to stdout / file.
- **Error handling:** wrap with `fmt.Errorf("...: %w", err)`. Use sentinel errors only for control flow (`var ErrParseSkipped = errors.New(...)`), not for messages. Test boundaries with `errors.Is` / `errors.As`. Never `panic` in production paths.
- **Config precedence (highest wins):**
  1. CLI flags (`--jobs`, `--file-timeout`, `--scan-timeout`, etc.)
  2. Environment variables (`AUDR_*`)
  3. `.audrignore` (suppression-only — does NOT set scanner config, only suppresses findings)
  4. Built-in defaults
  Future (v2): user-editable `audr.policy.yaml` slots between env vars and defaults.
- **Hardcoded ruleset (v1)** of 15-25 rules — see the inline list under "First v1 ruleset" below. Rules become editable YAML in v2.
- Output: HTML report (screenshot-friendly), SARIF v2.1.0 (drops into GitHub Code Scanning — must be spiked against GitHub's real importer in week 1 since SARIF for non-source-code findings has `physicalLocation` URI quirks), JSON (for piping). Every finding tagged `enforced | detectable | advisory` per the taxonomy above.
- **Offline by default** — no network calls during scan
- **Render hardcoded rules as YAML in the report** — CISO sees the inspectable rule list as if it were policy-as-code, even though v1 doesn't accept user-edited YAML
- **Suppression mechanism** — minimal `.audrignore` syntax (one rule-id per line, supports glob paths) plus inline `# audr:disable=rule-id` comment in matched files. Without this, every false positive becomes a support ticket.
- **Redaction at finding construction (defense in depth)** — secrets, env-var values, and credential-shaped strings are replaced with `<redacted:type>` *at the moment a Finding is constructed in `internal/finding/`*, NOT at output serialization. Parsers emit raw matches; the Finding constructor redacts before storing; formatters NEVER see raw secrets. This prevents leaks via `head`-piping, GHA log capture, build artifact upload, panic stack traces, debug logs, or any code path between parser and formatter. Critical because developers will screenshot reports and post to LinkedIn / Slack — leaking the secret Audr just found is the worst possible launch story.
- **Update path** — `audr update` checks GitHub Releases and downloads + verifies the next version. Manual fallback documented. Auto-update opt-in only.

### First v1 ruleset (15-25 rules)

The v1 ruleset must cover the founder's doc's "first 10 issues" list, expanded to ~20 once parser hardening reveals real-world patterns:

1. MCP server launched through unpinned `npx` (no `@version` suffix)
2. MCP server not in approved allowlist (allowlist is empty by default; rule fires only when configured)
3. MCP server receives env vars matching `PROD_*`, `*_PROD_*`, `STRIPE_LIVE_*`, `AWS_PROD_*`, `*_LIVE_*`
4. Agent config explicitly references `.env` or sources from broad secret files
5. Skill or rule file path is symlinked outside `~/.claude/skills/` or repo `./.claude/skills/`
6. Skill markdown contains shell command patterns matching `curl | bash`, `wget -O - | sh`, `eval`, base64-decoded payloads
7. Tool config combines shell + network + broad filesystem access on the same agent
8. GitHub Actions workflow grants `permissions: write-all` to a job that invokes an agent
9. Repo has agent instruction files (`AGENTS.md`, `CLAUDE.md`, `CODEX.md`) but no company policy file alongside
10. Local config (`~/.cursor/mcp.json`, `~/.claude/settings.json`) overrides repo-level config silently
11. Agent config exposes API keys in plaintext
12. MCP server uses `command: bash -c "..."` or shell pipelines as the launch command
13. Skill defines a tool that writes to `/etc/`, `~/.ssh/`, `/var/log/`, or other privileged paths
14. CI agent step has access to deploy tokens AND repository write
15. Agent config referenced through curl-fetched URL at runtime (dynamic config injection)

Add 5-10 more during parser hardening week.

**Targets to scan (priority order):**
- `~/.claude/`, `~/.cursor/`, `~/.config/Claude/`, `.mcp.json`, `.cursor/mcp.json` — enumerate every MCP server, command, args, env vars
- `.claude/skills/**/*.md` — flag skills with shell/network/WebFetch access; detect symlinks and external paths
- `.github/workflows/*.yml` — flag agent-invoking actions with `secrets.*` exposure
- `.env*`, shell rc files — detect secrets reachable by any discovered agent
- `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, `CODEX.md`, `.cursorrules` — flag rules that weaken security policy

### Scanner concurrency + robustness model

```
                                       ┌─────────────────┐
                                       │   walker        │  walks discovery roots,
                                       │   (1 goroutine) │  emits FilePath into chan
                                       └────────┬────────┘
                                                │  filePath chan (buffered N)
                                                ▼
            ┌───────────────────────────────────────────────────────────┐
            │                worker pool (size = GOMAXPROCS, --jobs N)  │
            │  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐    │
            │  │ worker1 │   │ worker2 │   │ worker3 │   │ worker4 │    │
            │  └─────┬───┘   └─────┬───┘   └─────┬───┘   └─────┬───┘    │
            │        │             │             │             │        │
            └────────┼─────────────┼─────────────┼─────────────┼────────┘
                     │   each worker per file:                 │
                     │   1. stat file (skip if >10MB)          │
                     │   2. ctx, cancel := context.WithTimeout(parent, 5s)
                     │   3. parse + apply rules under ctx      │
                     │   4. on parse error: log + emit "parse-error" finding, continue
                     ▼                                         ▼
                                  finding chan
                                       │
                                       ▼
                              ┌─────────────────┐
                              │   collector     │  aggregates findings,
                              │   (1 goroutine) │  applies suppression,
                              └────────┬────────┘  applies redaction,
                                       │           writes to formatters
                                       ▼
                              HTML / SARIF / JSON
```

**Defaults (overridable via flags):**
- Workers: `runtime.GOMAXPROCS(0)`, override `--jobs N`
- Per-file timeout: 5 seconds (`--file-timeout 5s`)
- Per-file size cap: 10MB (`--file-size-limit 10MB`); over-cap files emit a `parse-skipped:size` finding so the user knows
- Total scan timeout: 60 seconds (`--scan-timeout 60s`); first run on `~/` should typically complete in <10s
- Symlinks: detected and reported (rule #5 in the v1 ruleset). Default behavior: do NOT follow symlinks outside the scan root — that bounds I/O and is the safer security default
- Parse errors: log at WARN, emit `parse-error` finding tagged `advisory`, continue scan. Never crash on malformed input.
- Walk roots: the user's `$HOME` for machine-mode, the repo root for repo-mode. Respect `.gitignore` in repo-mode. Always honor `.audrignore`. **Always-skipped directories (defensive default)**: `node_modules/`, `vendor/`, `.git/`, `dist/`, `build/`, `target/`, `__pycache__/`, `.next/`, `.cache/`. These never contain agent configs and walking them tanks scan time on monorepos.
- HTML output: **inline everything** (CSS, severity icons as inline SVG, fonts via system stack). No external CDN. Required by offline-by-default; also makes the report screenshot-perfect on planes and air-gapped machines.
- Memory: bounded by `workers × max(file_size_cap)` ≈ 80MB on a default 8-core laptop. Predictable.

**Trust artifacts (the install-from-LinkedIn moat):**
- **Public source on GitHub.** License choice is open: BSL/FSL recommended for moat protection, MIT/Apache for procurement-ease. **Validate license choice with 2 design-partner legal teams before committing in code** — some procurement legal will reject non-OSI licenses; better to find out in week 2 than week 12.
- **Signed releases via cosign**, attached to GitHub Releases. Verify command in install script.
- **SBOM published** with every release (`syft` → SPDX or CycloneDX).
- **SLSA Level 2 provenance** (build provenance attestation via GitHub's official action). SLSA L3 (hermetic builds, non-falsifiable provenance) deferred to v2.
- **`audr self-audit` subcommand**. Defined semantics (industry-standard detached-signature pattern, same shape as `gh`, `syft`, `grype`, `trivy`):
  1. **Tamper detection via cosign detached signatures.** Locate the running binary's path. Look for `audr.sig` and `audr.crt` colocated. Verify the binary against the signature using sigstore's public-good transparency log (Rekor) — embedded verify logic, no `cosign` install required. If signature missing, mismatched, or revoked → loud red `selfAudit: TAMPERED` banner. *No self-baked hashes — that's a chicken-and-egg trap.*
  2. **Rule-set self-application.** Re-runs the same rule set on `audr`'s own source layout (its `.github/workflows/`, its own dependency list, its install script) and prints a green-or-red summary. The scanner that finds risky configs in others' repos must be clean in its own.
  3. **Output:** a `selfAudit: clean (cosign-verified)` block in every report header. If either tamper-check or rule-self-application fails, the block says so loudly and the report exits non-zero.
- **`audr verify <binary-path>` mini-tool** — embedded in the binary itself, can verify signature without requiring `cosign` on the user's machine. This solves the cosign-recursion-of-trust problem at install time.
- **No telemetry by default.** Opt-in only via `--share-anon` flag.

**Install motion:**

## Distribution Plan

**Binary distribution (free OSS CLI):**
- GitHub Releases with signed `.tar.gz` artifacts for darwin-amd64, darwin-arm64, linux-amd64, linux-arm64. Windows added in v2.
- One-line install: `curl https://audr.dev/install.sh | sh` — script uses the embedded `audr verify` to check signature, does NOT require pre-installed `cosign`
- Homebrew tap (`brew install audr/tap/audr`) once stable
- SBOM (SPDX or CycloneDX) published alongside every release
- SLSA L2 build provenance attestation
- Reproducible build instructions in `BUILD.md`
- **No npm/npx wrapper.** Adds Node supply chain to a tool whose entire pitch is "no node_modules." Direct contradiction with the trust thesis — explicitly rejected.

**SaaS platform (paid):**
- Hosted at audr.dev (or chosen domain). Cloudflare in front; a small Go API + Postgres backend; OAuth (GitHub + Google) initially, SSO/SAML in higher tier.
- CLI authenticates via API key; uploads scan results when `--sync` flag is passed.
- Pricing pages, docs, and signup are public; dashboard requires login.

**CI/CD pipeline for OSS releases:**
- GitHub Actions workflow on tag push: cross-compile binaries (darwin-amd64, darwin-arm64, linux-amd64, linux-arm64), generate SBOM via `anchore/syft-action`, sign binaries + checksums via cosign (`sigstore/cosign-installer` + `cosign sign-blob`), generate SLSA L2 provenance via `slsa-framework/slsa-github-generator`, attach all artifacts to the GitHub Release, update Homebrew tap formula.
- **No npm/npx wrapper publish step** — explicitly rejected, contradicts trust thesis (see "Cut from v1" above).
- Reproducible build verification job runs the build twice (or on a separate runner) and fails the release if SHA-256 hashes don't match. `-trimpath` and `-buildvcs=false` Go flags required for determinism. Lock the Go toolchain version per release.

**CI/CD pipeline for SaaS:**
- Standard cloud deploy (Fly.io, Render, or Vercel for the marketing site + Fly.io for the API). Defer choice until SaaS work begins.

**Distribution metrics to watch:** install count, signature verification success rate (should be ~100%), self-audit pass rate (should be 100% — anything else is an incident), HTML report screenshot share rate.
